# Session 35 — 2026-04-28 — Provider Resilience Hotfixes (v0.7.4)

**Scope:** Análisis de sesión `tui_0e91248e` (639 mensajes, 3h de uso intensivo). Tres bugs críticos encontrados y corregidos en cadena de failover de providers: DeepSeek V4 → GLM → MiniMax.

### Diagnóstico de la Sesión

Flujo: DeepSeek V4 Flash OK → Error 400 (`reasoning_content`) → Cambio manual a GLM → GLM deja de responder sin error (stream zombie) → Cambio manual a MiniMax → OK.

### Fix #1: DeepSeek V4 reasoning_content 400

**Causa:** DeepSeek thinking mode requiere `reasoning_content` en TODOS los mensajes `assistant` posteriores a un tool call. B-KODE tenía `_reset_tool_flag()` y `_has_had_tool_calls` escritos para esto pero nunca cableados — código zombie.
**Fix:** `_ensure_deepseek_reasoning()` parchea los mensajes antes de enviarlos a DeepSeek. El flag se activa en `_handle_tool_calls()` y se restaura al cargar sesiones.

### Fix #2: Streaming Timeout (Silent Hang)

**Causa:** El bucle `async for` sobre `chat_stream()` en `agent.chat()` no tenía timeout. Si el SSE stream del provider deja de emitir sin `[DONE]`, el bucle se cuelga para siempre — el TUI muestra "Thinking..." sin error.
**Fix:** `_stream_with_timeout()` envuelve el iterador con `asyncio.wait_for(stream.__anext__(), timeout=60s)`. Timeout por chunk (no total) — respeta streams largos, solo aborta zombies.

### Fix #3: Cloud API Polling Storm

**Causa:** `_poll_router_info()` llamaba a `get_router_info()` → `/v1/models` cada 5s para TODOS los providers. Contra APIs cloud (DeepSeek, MiniMax, Z.ai) esto es inútil (no exponen métricas de llama.cpp) y costoso (~720 req/hora por provider).
**Fix:** `ProviderClient.is_local` detecta si es localhost. `_poll_router_info()` hace skip si `not client.is_local`. Solo el router local recibe polling.

### Resultados

- 130 tests pasan sin regresiones
- 3 bugs corregidos en 3 archivos: `agent.py`, `client.py`, `tui.py`

# 2026-04-29 — Skills System v2.0: Vendor + User + BytIA Layers

## Contexto

BytIA-KODE era independiente pero ahora integra con el ecosistema BytIA. Surgió la necesidad de:

1. **Usuario nuevo**: instala KODE, tiene skills funcionando sin ~/bytia/
2. **Usuario con BytIA**: instala KODE, puede compartir skills con otros assistants
3. **Dual-mode**: KODE funciona standalone o integrado

## Decisión: Arquitectura de Capas

```
~/.bytia-kode/skills/
├── bytia/      # ~/bytia/skills/ (si existe) — prioridad más alta
├── user/       # Skills propias del usuario — writable
└── vendor/     # Skills core incluidas con KODE — read-only
```

Prioridad: **bytia > user > vendor**

## Implementación

### SkillLoader Reescrito

```python
class SkillLoader:
    def get_layer_dirs(self) -> list[tuple[Path, str]]:
        layers = []
        
        # 1. bytia (si existe ~/bytia/)
        bytia_dir = self.bytia_home / "skills"
        if bytia_dir.exists():
            layers.append((bytia_dir, "bytia"))
        
        # 2. user (siempre writable)
        user_dir = self.skills_home / "user"
        if user_dir.exists():
            layers.append((user_dir, "user"))
        
        # 3. vendor (siempre, bundled)
        vendor_dir = self.skills_home / "vendor"
        if vendor_dir.exists():
            layers.append((vendor_dir, "vendor"))
        
        return layers
```

# Session 37 — 2026-04-29 — Session Audit Fixes (v0.7.7)

**Scope:** Auditoría de sesión `tui_cf2c840b` (39 msgs, 2 min, agente GLM). Detectados 8 bugs, 5 fixeados + 1 flaky test corregido. 144 tests pasando.

### Problema

Sesión `tui_cf2c840b` mostró al agente repitiendo comandos bash bloqueados por security policy hasta 7 veces. Diagnóstico reveló Tool Error Memory con hash inestable.

### Root Cause

1. **Hash incluye JSON completo** — `_get_tool_error_key()` hasheaba `{"command": "...", "workdir": "...", "timeout": 120}` completo. Mismo comando con diferente `workdir` = hash diferente = bypass del memory.
2. **`df` no en allowlist** — Comando read-only esencial para diagnóstico requería workaround con `python -c`.
3. **Error messages sin hints** — "command not allowed" sin sugerir alternativas. Agente aprendía por ensayo-error (7 intentos).
4. **`testpaths` no configurado** — `pytest tests/` solo recogía 116 de 142 tests. 26 tests en `src/tests/` ignorados.
5. **Flaky test por global sucia** — `test_file_write_tool_handles_relative_path` no reseteaba `_WORKSPACE_ROOT`, contaminado por tests previos.

### Fix

| Fix | Archivo | Cambio |
|-----|---------|--------|
| Hash normalizado | `agent.py:690` | Hash solo `arguments["command"]` o `arguments["path"]` |
| `df` en allowlist | `registry.py:43` | Añadido `"df"` a `_DEFAULT_BINARIES` |
| Hints en errores | `registry.py:200` | Lista de permitidos + hint contextual para `cd` |
| pytest testpaths | `pyproject.toml` | `[tool.pytest.ini_options]` con `testpaths` |
| Flaky test fix | `test_basics.py:53` | `set_workspace_root(tmp_path)` antes de FileWriteTool |
